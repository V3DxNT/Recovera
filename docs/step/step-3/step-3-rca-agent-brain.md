# Step 3: Root Cause Analyzer (RCA) Agent Brain

This document details the architecture and implementation of the Root Cause Analyzer (RCA) pipeline, which upgrades the Agentic AI Brain to utilize a strict, machine-actionable schema for diagnostics and automated routing.

## Overview

The goal of Step 3 is to provide a rigorous, context-aware analysis pipeline. It bridges the initial detection (Step 2) with the forthcoming automated code remediation (Step 4). By establishing a structured JSON schema, we ensure that the LLM reliably produces discrete identifiers (`likelyFiles`, `likelySubsystem`) and actionable deterministic routing directives (`recommendedAction`).

## Core Components & File Functionality

### 1. Schema & Contracts
**`client/Agentic-AI/agent/types.ts`**
- **Functionality**: Defines the core TypeScript interfaces across the pipeline.
- **How it works**: We replaced legacy, loosely-typed fields (`root_cause`, `action_taken`, `reasoning`) with explicit machine-actionable fields. It defines `AgentOutput` consisting of `rootCauseSummary`, `failureMechanism`, `likelySubsystem`, `likelyFiles` (with path, reason, and confidence), `fixStrategy`, and a strict `recommendedAction` (using the `ActionType` union: `generate_fix`, `rollback`, `human_only`, `alert_only`, `unknown`).

### 2. Prompting & Parsing
**`client/Agentic-AI/prompts/system-prompt.ts`**
- **Functionality**: Instructs the LLM on exactly how to behave and format its output.
- **How it works**: Embeds the JSON schema into the system instructions. It forces the LLM to abandon conversational replies and output purely structured JSON containing the specific fields defined in `types.ts`.

**`client/Agentic-AI/agent/output-parser.ts`**
- **Functionality**: Validates the raw string response from the LLM.
- **How it works**: It strips markdown fences (e.g., ` ```json `), parses the string via `JSON.parse`, and runs it through a Zod validation schema. It clamps the confidence scores (between 0.40 and 0.93 for safety), enforces the array constraints on `likelyFiles`, and coerces any rogue `recommendedAction` strings into the `"unknown"` fallback. If validation fails, it returns a `ParseError` object to be handled safely by the decision engine.

### 3. Decision Routing & Safety Enforcement
**`client/Agentic-AI/tools/safety-registry.ts`**
- **Functionality**: Classifies `ActionType` values into safety brackets (`safe`, `needs_approval`, `blocked`).
- **How it works**: Maps the new strict RCA actions:
  - `generate_fix` / `alert_only` → `safe`
  - `rollback` → `needs_approval`
  - `human_only` / `unknown` → `blocked`
- **Importance**: The `decision-engine.ts` uses this classification (alongside the LLM's confidence score) to determine if an action can be auto-executed (`auto_fix`), if it requires human intervention (`approval_required`), or if it should just log a warning (`alert_only`). Mismatches here will inadvertently block all automation.

### 4. RCA Assembler Pipeline
**`client/lib/ai/rootCauseAnalyzer.ts`**
- **Functionality**: The central orchestration script that constructs the context for the agent and persists the result.
- **How it works**: 
  1. **Context Gathering**: Queries the database for the targeted incident and its associated events (`IncidentEvent`) to extract raw logs.
  2. **Dependency Stubbing**: Prepares clean insertion boundaries for future Steps (e.g., `fetchDeploymentHistory()`, `fetchRelevantCode()`, `findHistoricalIncidents()`). Currently, these return static mock data or `null`.
  3. **Invocation**: Formats this context into an `AgentInput` payload and calls `runAgent()` from the AI module.
  4. **Persistence**: Takes the final `AgentOutput`, determines the next version number, and stores the JSON-serialized result in the database via `prisma.incidentRca.create()`. It also updates the incident's `confidence` score.

### 5. API & Persistence Layer
**`client/prisma/schema.prisma`**
- **Functionality**: Database relational modeling.
- **How it works**: Added the `IncidentRca` model, establishing a one-to-many relationship with `Incident`. It stores the `rcaPayload` as a string/JSON alongside a `version` integer, enabling historical tracking if an incident is analyzed multiple times.

**`client/app/api/incidents/[id]/analyze/route.ts`**
- **Functionality**: The REST endpoint exposing manual RCA triggers.
- **How it works**: Resolves the dynamic `id` param asynchronously (complying with Next.js 15+ patterns), verifies the incident exists, and invokes the `analyzeRootCause` orchestrator from `rootCauseAnalyzer.ts`. Returns the structured output or an error payload to the dashboard.

## Next Steps Preparation
With the RCA pipeline fully validating its outputs and stubbing its context injection points, we are prepared for **Step 4: Automated Code Retrieval & Fix Generation**. The `likelyFiles` property generated by the RCA will directly feed into a vector database lookup, allowing the system to execute RAG over the repository and synthesize actual code patches (`fixGenerator.ts`).
